# 贡献与开发约束

本文件是人工开发者和 AI Agent 共用的最小开发契约。目标是在实现阶段遵守项目规则，同时避免在每次本地提交前重复执行远端 PR 的完整工作流。

## 何时读取

只有任务需要修改仓库文件时，才在第一次编辑前读取本文件。同一会话中已经读取且没有切换改动范围时，不要重复读取。

只读审查、代码解释、日志分析、方案讨论，以及不写入仓库的绘图或内容生成，不需要进入开发约束流程。

## 开发前

1. 检查 `git status`，保留已有未提交工作。
2. 阅读即将修改的实现、相邻代码和已有测试。
3. 识别改动范围：前端、后端/runtime、跨层、构建/CI、文档或资源。
4. 在开始实现前明确适用约束和计划执行的针对性检查。

不要为了规避失败而引入 `any`、禁用 lint、压制类型错误、删除测试或扩大无关改动。

## 开发中

- 遵循现有模块结构、命名、类型和错误处理方式，优先复用已有能力。
- 新功能应考虑对应测试；Bug 修复应考虑回归测试。
- 跨层修改应检查完整的数据流、类型、错误传播和兼容性。
- 每完成一个有意义的实现阶段，只运行与改动范围相关的检查。
- 不要在每次编辑、每个生成文件或每个 Agent 子任务后运行全仓验证。

常见范围检查：

- 前端：前端类型、相关前端测试和必要 lint。
- 后端/runtime：根类型检查和相关后端测试。
- 跨层行为：相关前后端测试及必要 E2E。
- 构建/CI：对应构建命令和验证套件。
- 文档/资源：格式、路径、命名、引用和必要构建检查。

## 本地提交

普通本地提交只需要：

1. Review 本次 diff，确认没有混入缓存、日志或无关文件。
2. 对修改范围执行必要的格式、lint、类型和相关测试。
3. 使用符合 Conventional Commits 的提交信息。

不要因为创建普通本地提交而默认执行 `npm run ci`、`npm run verify.all` 或全部平台验证。

## Pull Request

创建或更新 PR 后，由现有 GitHub Actions 和分支保护规则执行完整验收。远端工作流是 PR 是否可合并的最终依据。

推荐流程：

1. 推送开发分支并创建或更新 PR。
2. 查看 PR checks 状态。
3. 只读取失败 job 的日志，不重复加载成功任务的完整输出。
4. 针对失败项修改并运行相关本地检查。
5. 推送修复，由远端工作流自动复跑。

只有以下情况才考虑主动运行本地全量检查：

- CI 当前不可用。
- 修改了 CI、构建、测试框架或发布流程。
- 准备正式发布。
- 用户明确要求本地完整验证。

## Harness 入口

所有 Agent Harness 应注入同一条最小规则：

> 如果任务需要修改仓库文件，必须在第一次编辑前读取 `CONTRIBUTING.md`，识别改动范围并遵守对应约束。只读任务和不写入仓库的生成任务无需读取。同一会话未切换范围时不要重复读取；开发中只执行针对性检查，完整 PR 验收默认交给远端 GitHub Actions。
