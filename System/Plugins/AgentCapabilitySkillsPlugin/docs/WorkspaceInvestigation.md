# WorkspaceInvestigationSkill

## 简述

用于本地工作区代码、配置、提示词、UI 文案和错误处理的调查。目标是形成可验证的代码证据，而不是凭关键词或搜索摘要下结论。

## 何时使用

当用户询问“在哪里实现”“现在逻辑怎么样”“为什么这个行为发生”“某个 UI/配置/上下文怎么构建”，并且答案依赖本地仓库事实时使用。

## 不要使用的情况

用户只需要常识解释、架构头脑风暴或已经提供足够代码片段时，不要为了形式化流程重复搜索。

## 执行指引

1. 先判断线索类型：精确字符串、符号名、路径片段、错误文本、UI 描述、自然语言功能描述。
2. 有明确字符串或标签时优先精确搜索；有函数、类型、组件名时优先符号搜索；只有自然语言线索时用混合搜索。
3. 搜索结果只算候选。必须读取最相关候选文件的行范围后，才能说明代码行为。
4. 可用 ShellCommandTool 执行当前平台的只读片段检查、rg、目录观察或组合诊断；需要结构化证据时继续使用 WorkspaceReadFile/WorkspaceGrep。
5. 读取后根据 import、导出符号、调用方、组件层级、配置键、模板名继续检索，直到能解释因果链或确认缺口。
6. 多路径或长链路任务由 Pi substrate 负责保持执行循环，Senera 技能只提供任务组织和证据标准。
7. 如果候选互相冲突，优先使用最新读取到的代码事实，并说明仍缺少哪类证据。
8. 输出结论时引用已读取文件路径、行号或证据 ref。不要把搜索 snippet 当成完整上下文。

## 证据标准

结论需要至少包含一个已读取文件证据；涉及跨模块调用时需要读到调用方或被调用方之一。修改前应确认目标文件和相关约束。

## 完成标准

回答应说明：核心位置、实际逻辑、为什么这样判断、仍需验证的点。若要继续实现，下一步应明确编辑目标和验证命令。
