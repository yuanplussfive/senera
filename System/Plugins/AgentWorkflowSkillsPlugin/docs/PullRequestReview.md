# PullRequestReviewSkill

## 简述

用于审查 PR、合并后的变更、未提交 diff 或用户要求“看看有没有害”的改动。重点是风险、回归、测试缺口和框架影响。

## 何时使用

当用户要求查看 PR 做了什么、是否影响现有框架、是否需要提交、是否有残留文件或是否有 bug 时使用。

## 不要使用的情况

用户明确要求实现功能时，不要只停留在审查；用户只是询问概念时，不需要读取 diff。

## 工作流

1. 先确认变更范围：已合并提交、当前工作区 diff、指定文件或 PR 分支。
2. 先看文件列表和高层意图，再读关键文件的变更上下文。
3. 用户要求子代理、并行审查或多角色分工时，先用 AgentDelegateTool 展开 ParallelPullRequestReview。
4. 审查按严重度排序：行为回归、数据丢失、安全风险、并发/取消问题、测试缺口、可维护性。
5. 每条发现必须落到文件位置或可复现条件；没有明确问题时直接说明未发现阻断项。
6. 不要把“代码风格不同”当成问题，除非它会造成实际维护或行为风险。
7. 如果用户要求提交，排除无关本地改动和残留日志，再提交本次相关改动。

## 证据标准

审查结论需要来自实际 diff 或已读取文件。测试建议要对应到变更风险，而不是泛泛要求“加测试”。

## 完成标准

输出优先是发现列表；没有问题时说明无发现，并补充剩余风险或未运行的验证。
