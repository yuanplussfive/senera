# Senera 升级指南

Senera 的桌面端、服务端和容器从同一个 Git 标签构建。正式版本使用 `vX.Y.Z`，安装包内部版本、GitHub Release 和容器镜像版本保持一致。

## 升级前

1. 关闭正在运行的 Senera 桌面端和服务端。
2. 备份当前工作区内需要保留的项目文件。
3. 备份 Senera 用户数据目录，尤其是配置数据库、会话记录和自定义插件配置。
4. 阅读目标版本的 [更新记录](../CHANGELOG.md)，确认是否存在破坏性变化。

桌面端用户数据目录由 Electron 按操作系统管理。Windows 默认位于 `%APPDATA%\Senera`。如果在启动参数或系统策略中修改过用户数据目录，应备份实际使用的位置。

## 桌面端升级

从 GitHub Release 下载与目标版本一致的 `SeneraSetup-X.Y.Z.exe`，退出旧版本后运行安装程序。安装程序升级应用文件，不应依赖覆盖工作区内容。

首次启动后检查：

- 配置是否能够正常读取；
- 历史会话是否可以打开；
- 自定义插件是否成功加载；
- 沙箱状态是否符合当前机器的虚拟化能力。

## 容器升级

生产环境应固定完整版本，不要只依赖 `latest`：

```yaml
image: ghcr.io/<owner>/senera:1.2.3
```

拉取目标版本后重新创建容器。配置和持久数据应通过 volume 保存，不应写入容器可写层。确认新版本健康后再清理旧镜像。

## 版本迁移

`desktop-v1.0.21` 是旧桌面发布体系的最终基线。新体系从根包版本 `1.0.21` 继续推进，并使用标准 `vX.Y.Z` 标签。旧标签仍可用于定位历史安装包，但不会继续增长。

版本变化遵循 SemVer：

- patch：兼容的问题修复；
- minor：向后兼容的新功能；
- major：需要人工处理的破坏性变化。

如果某个版本需要修改配置、数据库或插件清单，对应步骤必须同时写入本文件和该版本的 CHANGELOG，不能只记录在提交信息中。

## 回退

1. 停止当前版本。
2. 恢复升级前备份的用户数据。
3. 桌面端重新安装上一版本；容器重新指定上一完整版本标签。
4. 启动后验证配置、会话和插件状态。

不要让旧程序直接读取已经被新版本执行过不可逆迁移的数据。存在不可逆迁移时，发布说明必须明确标注，并要求通过备份回退。
