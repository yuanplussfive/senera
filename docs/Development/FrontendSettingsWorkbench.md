# 前端设置工作台约束

`SettingsWorkbench`（设置工作台）是桌面端、Web 端和响应式布局共用的主设置界面。新增设置能力时，优先扩展这个界面；旧弹窗只能作为临时 fallback（兜底路径），不要再新增一套独立的业务设置界面。

## 贡献约束

- Electron、Web 和响应式视图必须共享同一套设置业务逻辑。宿主侧代码可以负责打开窗口、同步路由或调整尺寸，但不能分叉供应商、模型、技能或系统配置行为。
- 每个 SettingsWorkbench 工作区内部只能有一个主纵向滚动面。嵌入式的供应商、模型、系统和技能分区不要再套会截获页面滚动的内部详情列表滚动。
- 已迁移分区必须接入 `Frontend/src/features/settings/settingsPresentation.ts` 中的共享展示模型。规划中分区只有在能加载数据、完成必要编辑或查看、展示错误并通过人工检查后，才可以启用。
- `主配置`、运行、规划、检索和存储继续共用配置草稿和保存流程；`模型服务` 与 `默认模型` 使用各自的供应商/模型命令即时保存。用户导航不再提供 `providers` 或 `models` 入口；后端表单内部仍可保留名为 `models` 的数据 schema，供模型服务读取模板和快照。
- 技能和插件设置必须走插件操作流程，不能混入主配置草稿保存流程。
- 状态、禁用原因、dirty state（未保存状态）和保存归属必须可见。如果保存或刷新按钮不可用，界面要说明原因。
- 兼容弹窗要保持轻量。某个分区迁移后，新行为应写进 SettingsWorkbench 复用的内容组件，而不是写进旧弹窗壳。

## 测试约束

- `settingsPresentation.test.ts` 覆盖分区分组、搜索详情、迁移状态和运行态摘要。
- `settingsInteractionModel.test.ts` 覆盖草稿状态、保存禁用原因和插件运行态。
- 如果组件改动影响滚动归属、保存语义或分区启用条件，同一个改动里要同步更新这些测试。

## 人工检查

修改 SettingsWorkbench 行为并提交前，需要检查：

1. 打开 `主配置`、`模型服务`、`默认模型` 和 `技能`。
2. 内容溢出时，确认工作区卡片作为一个整体滚动。
3. 确认未保存状态、保存、还原或刷新、加载和错误信息仍然可见。
4. 确认 Chat fallback 仍可使用旧模型配置组件，但 SettingsWorkbench 不再暴露旧 `providers/models` 路由或重复默认模型入口。
