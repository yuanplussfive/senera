# 前端设置工作台约束

本文定义 Senera 设置工作台的产品表面、信息架构、交互语义、响应式布局与实现边界。`SettingsWorkbench` 是设置业务的唯一完整界面；Electron 与 Web 可以使用不同宿主，但不得分叉设置内容、保存协议或领域行为。

## 1. 产品定位

设置是主工作区上的辅助任务，不是独立的管理后台。

- 用户进入设置时，应能明确感知自己仍处于当前 Senera 工作上下文中。
- 设置内容使用 Senera 自己的产品术语，不按后端 schema、迁移阶段或实现模块组织界面。
- Electron、Web、平板和移动布局共享同一套设置分区、表单语义、校验规则和领域命令。
- 宿主只负责打开方式、尺寸、历史记录、焦点与关闭行为，不拥有设置业务。

## 2. 设置宿主

### 2.1 Web

Web 端从应用内点击“设置”时，必须在当前主工作区上方打开大型模态工作区，不得通过页面跳转替换整个 `App`。

- 主会话、滚动位置和工作流上下文必须保留。
- 设置打开期间，背景工作区降为非交互层并停止页面滚动。
- 点击背景遮罩不得关闭设置。
- 设置模态必须提供明确的关闭按钮。
- 小屏下设置模态转为全屏工作区，不保留外围边距、模态圆角或可点击遮罩区域。
- Web 设置必须复用主应用已有的连接和运行时数据，不得因为打开设置而创建第二套应用根节点或重复 WebSocket。

### 2.2 Electron

Electron 继续使用独立 `BrowserWindow` 承载设置。

- 独立设置窗口与 Web 模态共享同一个 `SettingsWorkbench`。
- Electron 宿主可以独立初始化连接、窗口尺寸和窗口生命周期，但不得复制设置业务组件。
- 不要求 Electron 与 Web 使用相同打开形态；要求两者具有相同的信息架构、保存语义和操作结果。

### 2.3 唯一业务界面

`SettingsWorkbench` 是唯一完整设置界面。

- 不得为 Web、Electron、移动端或某个入口创建另一套简化设置页。
- 页面入口不得直接决定具体宿主；入口向应用层发出“打开某个设置分区”的意图，由应用层选择 Electron 窗口或 Web 模态。
- 与当前任务强相关的快捷控件可以存在，但不能扩展成第二套完整设置界面。

## 3. Web URL 与浏览器历史

Web URL 表达“当前工作区上方打开了哪个设置分区”，而不是切换到另一个应用页面。

- 从应用内打开设置时，使用一次 `pushState` 增加设置打开状态。
- 切换设置分区时更新 URL，但使用 `replaceState`，不得为每次分区切换堆积历史记录。
- 浏览器返回用于请求关闭整个设置工作区。
- 刷新设置 URL 或直接访问深链接时，必须加载主应用并自动打开目标设置分区。
- 直接访问设置深链接后关闭设置时，应移除设置参数并保留主应用，不得退出应用或进入空页面。
- URL 中不存在或不可用的分区必须回退到第一个可用导航项。

推荐 URL 语义：

```text
/chat/session-123
/chat/session-123?settings=appearance
/chat/session-123?settings=model-service
```

## 4. 导航与信息架构

### 4.1 分组

导航必须按照用户可理解的产品领域分组，不得将所有分区平铺为同一级列表。

推荐分组：

```text
模型
  模型服务
  默认模型

能力与运行
  运行
  规划与学习
  检索与向量
  技能

个人
  通用
  外观

系统
  文件与界面
  关于
```

- 分组名称描述用户控制的领域，不描述代码模块。
- 尚未形成完整能力的分区不得进入正式导航。
- 正式导航不得显示“迁移中”“规划中”等开发进度。
- 不得为了补齐参考产品的栏目而新增空设置页。
- 将来新增设置分区时，应先确定所属用户领域和交互契约，再加入导航。

### 4.2 导航视觉

导航参考紧凑、低干扰的设置侧栏表达：

- 分组标题使用弱化文字，与导航项形成清晰但克制的层级。
- 导航项保持单行、紧凑行高和稳定的左右内边距。
- 图标只承担识别作用，不使用彩色图标容器或装饰性底板。
- 当前项使用整行浅色背景，不使用厚重卡片、高饱和填充或常驻状态胶囊。
- 设置搜索位于导航上方，并保留分组结构；搜索结果不得退化成无法判断所属领域的无序列表。
- 导航区域可以独立滚动，但标题、搜索和当前分组结构必须保持可理解。

### 4.3 默认分区

- 通用“设置”入口默认选中第一个可用导航项。
- 默认分区不得通过另一份固定常量重复写死，应从最终可用导航顺序推导。
- 明确指定分区的入口和 URL 深链接可以覆盖默认选择。
- 不记录或恢复“上次访问的设置分区”。
- 第一个分区被禁用或移除后，自动选择下一个可用项。

## 5. 设置交互契约

每个设置分区必须明确属于以下三种契约之一。不得自行发明含义不清的第四种保存方式。

### 5.1 即时偏好

适用于外观、通用布局和动画偏好。

- 修改后立即应用。
- 不显示页面级“保存”按钮。
- 可以提供“恢复默认”或明确的撤销操作。
- 关闭设置时不触发未保存确认。
- 正常修改不需要反复显示成功 toast。

### 5.2 共享配置草稿

适用于运行、规划与学习、检索与向量、文件与界面。

- 这些分区共享同一份配置草稿和保存协议。
- 用户可以在相关分区之间切换，草稿继续保留。
- 有修改的相关分区可以使用克制的未保存提示，但不得恢复通用状态卡或常驻状态胶囊。
- 保存、还原、校验错误和保存失败必须有明确反馈。
- 关闭整个设置工作区时，未保存草稿必须进入统一关闭确认。
- 用户主动还原或确认放弃时才清除共享草稿。

### 5.3 实体管理

适用于模型服务、默认模型、技能以及后续同类管理界面。

- 新增、编辑、删除、重命名和选择默认项按实体单独确认和提交。
- 不纳入主配置的页面级“保存全部”。
- 尚未确认的编辑表单属于本地草稿。
- 离开当前实体或所在分区前，必须保护尚未确认的本地草稿。
- 已确认并成功提交的操作立即成为正式状态，不再进入关闭确认。
- 操作失败必须在相关表单或实体附近说明原因，toast 只能作为补充。

## 6. 关闭与未保存保护

关闭按钮、`Esc` 和浏览器返回必须进入同一个“请求关闭设置”流程。

- 没有未保存内容时立即关闭。
- 存在共享配置草稿或实体编辑草稿时，显示确认对话框。
- 用户确认放弃后清除对应草稿并关闭。
- 用户取消关闭后保持当前分区、滚动位置和输入内容。
- 即时偏好不参与未保存确认。
- 子级对话框打开时，`Esc` 只关闭最上层对话框，不得穿透并关闭整个设置工作区。
- 关闭后焦点必须返回触发设置的控件；直接链接打开时返回到主工作区的合理焦点位置。

## 7. 操作反馈

设置界面不得使用通用状态卡或迁移状态面板。

必须删除或禁止：

- 页面顶部的通用状态卡；
- “已同步”“已配置”“正常连接”等常驻成功状态；
- “迁移中”“规划中”等开发进度；
- 导航项后的常驻状态胶囊；
- 用于证明 DOM、连接字段或配置同步正常的内部状态摘要。

只保留完成当前操作所必需的反馈：

- 数据加载反馈；
- 输入校验错误；
- 保存或提交进行中；
- 操作不可用的实际原因；
- 保存、获取、删除等操作失败；
- 尚未确认或尚未保存的修改；
- 关闭时的草稿保护。

这些反馈必须靠近相关字段、表单或操作按钮，不得重新汇总成全局状态卡。正常状态应保持安静。

## 8. 大屏与响应式布局

### 8.1 Web 模态尺寸

- 大屏和超宽屏的设置模态最大宽度约为 `1440px`，居中显示，不随 2K 或 4K 屏幕无限拉伸。
- 普通桌面使用接近全屏的尺寸，推荐宽度范围为 `calc(100vw - 32px)` 至 `calc(100vw - 64px)`。
- 高度基于 `100dvh` 计算，并保留适当上下边距；不得使用会受移动浏览器地址栏影响的固定 `100vh` 假设。
- 小屏使用全屏设置工作区。
- 超宽屏增加的是稳定留白和可读性，不是无限拉长输入框、卡片或文本行。

### 8.2 分区宽度

不同分区必须根据内容类型采用不同宽度策略，不得共享一个固定 `max-width`。

- 外观、通用：居中，建议最大宽度约 `1080px` 至 `1160px`。
- 运行、规划与学习、检索与向量、文件与界面等配置表单：居中，建议最大宽度约 `1200px` 至 `1280px`。
- 模型服务、技能等列表—详情工作区：填满设置内容区域。
- 关于和诊断信息：居中，建议最大宽度约 `900px` 至 `1000px`。
- 所有受限宽度分区必须同时设置 `width: 100%` 和水平居中，禁止贴在左上角形成单侧巨大空白。

### 8.3 容器适配

设置内部布局优先根据设置工作台容器宽度判断，不得只依赖浏览器 viewport。

- 复杂分区使用容器查询或等价的容器宽度模型。
- 模型服务在空间充足时展示供应商列表与详情；空间不足时转为列表—详情页面切换。
- 全局设置导航是否常驻、模型服务是否双栏，以及详情区域宽度必须由实际可用空间决定。
- 响应式切换不得造成当前选择、草稿或滚动上下文丢失。

## 9. 滚动模型

禁止主要纵向滚动容器层层嵌套；允许主从工作区使用边界清晰的同级滚动区域。

- 设置导航可以拥有独立滚动区域。
- 外观、通用和配置表单等普通分区只有一个内容纵向滚动面。
- 模型服务、技能等全高主从工作区可以让左侧列表和右侧详情分别滚动。
- 主从滚动区必须是同级区域，不能在右侧详情的主要滚动面内继续嵌套另一个占据主要高度的滚动面。
- 移动端一次只展示列表或详情，当前页面拥有唯一主要滚动面。
- 下拉菜单、选择器和对话框中的短暂候选列表不属于主要工作区滚动面。
- 滚动时必须保持必要的保存、确认或返回操作可达。

## 10. 文案与诊断信息

面向普通用户的设置只描述用户能够理解和控制的能力。

不得在普通设置中直接展示：

- DOM token 或 DOM attribute；
- schema、内部字段名或命令名；
- `contract` 等实现术语；
- 原始同步状态；
- 迁移进度和开发验证线索；
- 仅用于证明内部实现正确的表格。

要求：

- 标签使用用户认知中的对象和动作命名。
- 按钮必须准确描述将发生的操作，例如“保存更改”“获取模型”“恢复默认”。
- 错误信息说明发生了什么以及用户可以如何处理，不能只暴露状态码。
- 对用户有实际排障价值的技术信息移动到“关于”中的诊断区域。
- 诊断信息默认折叠，并提供复制能力；纯开发信息可以只在开发环境出现。

## 11. 动效与无障碍

- Web 设置模态使用克制的透明度与轻微缩放过渡，打开约 `180ms` 至 `220ms`，关闭略快。
- 桌面分区切换不得使用大幅整页滑动；移动端列表—详情切换可以使用方向明确的页面过渡。
- 必须遵守 `prefers-reduced-motion`，减少动效时不得依赖动画表达状态。
- 模态必须正确限制焦点，子级对话框必须维持正确的焦点层级。
- 所有导航、关闭、保存、还原、确认和返回操作必须支持键盘。
- 当前导航项、错误、加载和不可用状态不能只通过颜色表达。
- 点击目标和控件密度需要同时适配鼠标与粗指针设备。

## 12. 实现边界

- 设置宿主、设置业务和设置运行时必须分层。
- `SettingsWorkbench` 不得读取浏览器位置、直接调用 Electron bridge 或自行决定关闭方式。
- 分区选择应支持受控状态，由宿主同步 URL、窗口入口和默认分区。
- Web 主应用与 Web 设置模态共享连接、配置快照和领域命令。
- Electron 独立设置窗口可以创建自己的运行时宿主，但必须使用相同命令接口。
- 分区组件不得复制供应商、模型、技能或配置保存协议。
- 响应式宿主变化只能改变布局和呈现方式，不能改变业务能力或保存语义。

## 13. 验收检查

修改设置工作台及其宿主后，至少检查：

1. Web 从当前会话打开设置时，主工作区没有卸载或丢失上下文。
2. Web 只建立并使用主应用已有的连接，没有因为打开设置创建重复 WebSocket。
3. 浏览器返回可以关闭设置，分区切换不会堆积多条历史记录。
4. 刷新和直接访问设置深链接可以恢复目标分区。
5. Electron 仍通过独立窗口打开相同的设置工作台。
6. 通用设置入口选择第一个可用导航项，明确入口和深链接可以覆盖它。
7. 设置导航按产品领域分组，不显示迁移进度或常驻状态胶囊。
8. 顶部没有通用状态卡，操作必需反馈位于相关控件附近。
9. 即时偏好、共享配置草稿和实体管理分别遵守自己的交互契约。
10. 共享配置草稿能跨相关分区保留；实体草稿离开编辑上下文前会得到保护。
11. 关闭按钮、`Esc` 和浏览器返回执行相同的草稿检查；遮罩点击不会关闭设置。
12. 在超宽屏、普通桌面、平板和移动宽度下检查内容宽度、滚动归属和列表—详情切换。
13. 外观和普通表单居中显示，模型服务等工作区合理使用可用宽度。
14. 主从分区不存在主要滚动容器层层嵌套的问题。
15. 普通用户界面不显示 DOM、schema、contract 或迁移实现语言。
16. 键盘焦点、关闭后焦点恢复、嵌套对话框和减少动效模式工作正常。
